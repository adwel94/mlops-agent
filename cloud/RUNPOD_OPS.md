# RunPod 운영 규칙

학습·평가로 RunPod GPU 를 사용할 때 읽어봐야 될 규칙.

> 과금 파드다. 명시 요청 시에만 띄우고, 끝나면 반드시 종료.

## 관측 — 이 스크립트로 상태를 본다

| 스크립트 | 볼 수 있는 것 |
|---|---|
| `cloud/common/read_logs.py` | 파드 raw stdout (부팅·다운로드·기동·에러) |
| `cloud/train/wandb_status.py` | 학습 loss/lr 등 지표 |
| `cloud/common/ai_report.py "msg"` | #ai-리포팅 채널로 내가 리포트를 보냄 |
| BOOT_TIMING 채널 | 부팅 단계별 소요 — "부트스트랩 왜 느리지" 과거와 비교 (`read_logs.py --webhook`) |

PIPELINE·RUNPOD 채널은 파드가 Discord 에 올린다(사람이 봄).

## 폴링

파드 상태는 자동 알림이 없어 직접 확인한다. `Bash` `run_in_background` 로
`sleep <N> && <확인>` 을 돌려 알림으로 결과를 받고 다음 수를 정한다. 한 턴에 한 번,
포그라운드 `sleep` 금지.

## 부팅 확인 (5분 간격)

- **재기동 조건 = 확인된 불가역 실패** — 부팅 로그에서 이 호스트가 우리 환경을 못 띄우는
  실패가 확인되면(GPU/드라이버 미탐지, 패키지·flash-attn·uv_sync 설치 실패 등) 즉시 종료하고
  `--cloud-type SECURE` 로 재기동한다. 그 전(부팅·다운로드·GPU 로드·5555 대기)에는 기다린다.
  느리거나 의심스러우면 재기동 대신 `ai_report.py` 로 사람에게 알린다(아래 진행 리포팅).
- **준비 신호** — `RUNNING` 은 아직 아님(다운로드·GPU 로드 남음). serve=5555 바인딩,
  train=지표·로그가 학습 단계로 진행.
- **받은 것 대조** — `read_logs.py` 로 fetch 로그를 보고 의도한 걸 받았는지 확인.
  예: 모델 다운로드 로그의 `@<revision>` 이 원한 태그인가(`@main` 이면 종료).
- **부팅 전략 — 최소 레이어 이미지 + 클라우드 부팅 시 설치.** 가장 가벼운 레이어 이미지
  (RunPod 공식 베이스와 거의 동일)를 쓰고 pip/uv 설치는 부팅 때 클라우드에서 한다 → 이미지
  pull 이 빠르고 GPU arch 호환성(flash-attn 등)도 확보된다(실험으로 검증됨). 이미지가 이미
  최소이므로 부팅 속도의 변수는 **호스트**다: **warm 호스트**(공식 베이스 레이어가 캐시됨)는
  1~2 분 내 stdout·5555 가 뜨고, **cold 호스트**는 베이스를 새로 받느라 더 걸린다. 컨테이너
  기동 신호(`pod()` 의 `publicIp`·`machine`·stdout)가 수 분째 비어 있으면 cold 호스트이니
  종료 후 재기동해 **warm 호스트를 잡는다**. (boot-install 소요는 BOOT_TIMING 채널로 과거와 비교.)

## 진행 리포팅 (20분 이하 간격)

돌아가는 동안 `ai_report.py` 로 #ai-리포팅 에 추세를 남긴다. 시작·변화·완료·이상은 필수, 도배 금지.

**이상·의심은 즉시 알린다 (정상 추세만 20분 간격).** 과금 파드가 조금이라도 이상하면(부팅이
예상보다 느림·응답 없음·호스트 의심 등) 혼자 판단해 삼키지 말고 `ai_report.py` 로 "이거 문제일
수 있으니 한번 봐줘" 한 줄이라도 즉시 보내 **사람을 루프에 넣는다** — 과금 중에 조용히 혼자
해결하려다 시간·비용을 날리는 게 최악이다.

## 연결

`publicIp` + 5555 매핑 포트 = `runpod_client.pod(pod_id)`. (`runpod_ls` 는 포트 안 보임.)

## 종료

완료·실패·중단 어느 때든 `/runpod_down <pod_id>`.
