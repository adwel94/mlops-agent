# AI 협업 피드백 — 진단 과정에서 드러난 추론 실패와 교정

이 문서는 "왜 모델이 저성능인가" 진단 중 AI(나)가 **틀렸다가 사용자 반박으로 정정된**
사례를 기록한다. 목적은 책임 추궁이 아니라 **같은 실패를 막는 지시 패턴**을 남기는 것.
기술적 진단 내용은 `model-eval-diagnosis.md` 참고.

## 1. 무슨 실수를 했나 (3건, 모두 사용자가 잡아냄)

| # | 내가 주장한 것 | 실제 | 정정 근거(내가 이미 갖고 있던 것) |
|---|---|---|---|
| ① | "Path A(절대 rot6d)는 end-to-end 검증 안 됨 = 갭" | **중복**. 이미 검증됨 | 저장 액션=`pose[t+1]`, `ee_verify` 타깃=재구성된 `pose[t+1]`(recon~0), rot6d round-trip~3e-7 → 같은 pose·같은 IK. ee_verify ~100%면 Path A도 ~100% |
| ② | "정규화/relative_stats 누락이 유력 결함" | **배제**. 자기일관 | 학습 정상완료(loss 0.03)+serve가 같은 체크포인트 로드 = 학습·추론 정규화 일관. 깨졌으면 학습이 안 떴음 |
| ③ | 진단이 "무과금" | **과금**. serve=GPU 파드 | `gr00t_eval`이 `server_host`(RunPod GPU)에 붙는 코드를 바로 직전에 읽음 |

## 2. 공통 뿌리

**이미 가진 증거로 내 주장을 반증해보지 않고 단정했다.** 세 번 다 정정 근거가 내가
*이미 읽은 코드/결과* 안에 있었는데, 그 한 칸의 검산(값 대입 / 의존 사슬 추적)을
사용자가 대신 했다.

배경 동기: 사용자가 "리스트 뽑아봐"라고 했을 때 **"많이 나열 = 좋은 답"**으로 오해하고
A~F를 깔았다. 그중 ①②는 가진 증거로 이미 배제됐어야 할 것들. → **breadth로 depth를
흉내**냈다. "이론상 가능"과 "현재 증거와 양립"을 안 갈랐다. 정확함이 아니라
"꼼꼼해 보임"을 좇은 것.

## 3. 나를 이렇게 지시하면 위 실패가 막힌다

1. **"리스트 말고, 제일 유력한 가설 하나 + 그걸 반증할 가장 싼 테스트만."**
   → 헷지성 나열 차단.
2. **"각 주장마다 *반증* 증거가 뭔지, 우리가 *이미 아는 것*으로 깨지는지 먼저 확인하고 말해."**
   → ①② 직격.
3. **"데이터/제어 흐름을 값 대입해서 끝까지 따라가고 나서 단정해."**
   → "갭"이라 부르기 전에 검산.
4. **"과금 여부는 코드가 어디서 도는지가 아니라 의존 사슬로 판단해."**
   → ③ 차단.
5. **"이미 가진 증거로 배제되는 후보는 빼고 말해."**
   → 죽은 가설 제거.
6. **사용자가 기술 근거로 2번 반박하면 = 내 확신 과함의 신호.** 우기지 말고 증거에 재검산.

## 4. CLAUDE.md 게이트와의 연결

- ②는 "모르면 결함 취급" 게이트를 **생각의 대체물**로 오용한 사례. 그 게이트는 "우회하기
  전에 멈춰라"지 "가진 증거를 무시하고 의심을 부풀려라"가 아니다. **멈추되, 가진 증거는
  먼저 다 쓴다.**
- 산출물 무결성 체크("이 단계가 약속한 걸 온전히 내놨나")는 **내 주장에도 적용**한다 —
  주장을 내놓기 전에 "이 주장이 우리가 아는 사실과 충돌하지 않나"를 self-check.
