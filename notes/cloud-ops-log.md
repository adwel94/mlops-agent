## 2026-06-30 — RunPod community 호스트 `CUDA unknown error`

**증상.** 평가용 정책 서버(serve)를 띄우다 **3대 중 2대**가 부팅 막판
`CUDA unknown error`(GPU는 붙어 있는데 torch가 devices=0 으로 못 잡음)로 죽었다.

**1차 대응 (transient 가정).** 간헐적 호스트 결함으로 판단 → `serve_policy.sh` 에
**재시도 루프**(죽으면 fresh process 로 다시 띄움, 4회)를 넣어 이미지 재배포. 이땐
정작 검증 부팅이 한 번에 떠서 재시도 경로가 실제로 발동되진 않았다.

**재발 & 원인 규명 (v2-s20000 평가 때 실제 발동).** serve 가 또 CUDA 에러로 죽었고
재시도 루프가 4회 돌았으나 **전부 같은 에러로 실패** — fresh process 로도 회복 안 됨.
파고드니 **RunPod community 스케줄러가 같은 불량 호스트(publicIp `60.249.37.148`)를
계속 재배정**(파드 3개 연속, 5분 대기로도 안 바뀜)하고 있었다.

**해결.** `--cloud-type SECURE`(다른 호스트 풀, 예: `103.196.86.42`) → 한 번에 정상 기동.

**교훈 (→ README 알려진 제약 으로 승격).**
- 재시도 루프는 *transient* 결함엔 듣지만 **호스트 영구 결함**엔 안 듣는다 → 호스트
  자체를 바꿔야 한다(Secure, 또는 부팅 전 IP 확인 후 재생성).
- 파드 생성 직후 `publicIp` 를 먼저 확인 — 직전 불량 호스트와 같으면 부팅 완료를
  기다리지 말고 즉시 종료해 과금을 아낀다.
- 2번 실패하면 같은 우회를 연쇄하지 말고 멈춰 보고(시도 예산) — 여기선 재시도 4회가
  같은 벽을 4번 친 꼴이었다. 호스트 풀을 바꾸는 게 진짜 수였다.
