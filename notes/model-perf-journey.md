# 모델 성능 여정 — "왜 학습한 로봇이 큐브를 잘 못 집었나"

> **작업 히스토리 (시점 기록).** 이 문서는 **하나의 축**만 말한다: v1 모델의 저성능을
> 관찰 → 진단 → 처방 → 검증까지 따라간 여정. 무엇을 의심했고, 어떻게 확인했고, 무엇을
> 알게 됐는지를 순서대로. 현행 지침이 아니라 그때의 조사 로그다.
>
> - 진단에서 얻은 추론 규율 → CLAUDE.md 협업 규칙 "추론 규율"로 승격.
> - 이 진단이 가리킨 수정(손목 카메라)을 **어떻게 구현·검증했나**(설계·코드·파이프라인)
>   → `wrist-camera-impl.md`. 여기(§5)는 그 **결과만** 싣는다.

## 한눈에 보는 결말

우리 로봇 모델은 **"빨강 큐브 집어"** 같은 지시를 받고 큐브를 집는 일을 한다. 그런데
성공률이 ~20%밖에 안 나왔다. 파고들어 보니:

- **색깔은 거의 안 틀린다** (집기만 하면 94% 맞는 색) — 언어 이해는 문제 아님.
- 진짜 문제는 **손을 큐브까지 정밀하게 못 가져간다** — 평균 8cm 빗나가서 헛손질.
- 측정 오류도 데이터 불량도 아닌 **진짜 모델 한계**임을 (공짜 검증으로) 확인.
- 원인: 모델의 **"눈"이 얼어 있어서** 큐브의 정확한 3D 위치를 못 짚고 + **단일 고정 카메라**라 깊이가 애매하다.
- 처방: 처음엔 **눈 녹여 재학습**(visual fine-tune, B)을 1순위로 봤으나, 공식 정밀 레시피
  재검토에서 **손목 카메라 추가**(D)로 재정렬 — NVIDIA/Seeed 표준이 손목+정면 2캠이다.
  **D를 채택·완료: 손목캠 효과 검증됨** — 같은 학습량에서 1캠 0.32 → 손목캠 0.54,
  빗나감 8.4cm→1.5cm (§5).

## 0. 먼저, 우리가 뭘 만들었나

테이블 위에 빨강·초록·파랑 큐브 3개가 있고, **"○○색 집어"**라고 시키면 그 색 큐브를
집어 드는 로봇 팔. 이걸 GR00T라는 거대 AI 모델로 학습시켰다.

**채점(평가) 방법**: 학습한 모델을 클라우드 GPU에 올려두고(serve), 내 PC의 시뮬레이터에서
로봇을 굴리며 모델에게 "다음에 어디로 움직일까?"를 물어본다. 모델이 답한 손 위치를
관절 각도로 바꿔(IK) 실제로 움직이고, **시킨 색 큐브를 집어 들면 성공**으로 친다.

기준점 하나: 원래 시범 데이터(사람이 만든 정답 궤적)는 이 채점에서 **100% 성공**한다.
그러니 모델이 그보다 낮으면 그만큼이 모델의 부족분이다.

**과금 구조** (진단 시작 전 정정한 것) — 정책 서버(모델 추론)는 RunPod GPU라 **과금**,
롤아웃(sim·IK·계측)은 로컬 WSL이라 무과금. 그래서 "코드 보강은 지금, serve 파드는 별도
승인 후"로 갈랐다.

## 1. 문제: 성공률이 20%밖에 안 나온다

학습을 길게(스텝 20,000) 시켰는데도 성공률 ~18%. 짧게(3,000) 시킨 것과 **똑같았다.**
이상한 단서가 셋이었다:

1. **학습 오차(loss)는 거의 0까지 떨어졌다** (1.06→0.03) → 모델이 배울 건 다 배웠다.
2. **로봇이 "갈 수 없는 자세"를 요청한 적은 한 번도 없다** (`ik_fails=0`) → 동작이 미친 건 아님.
3. **더 오래 학습해도 똑같다** (3k=20k=18%) → "학습이 부족해서"는 아님.

제약을 한 줄로: `orig_success=50/50 · policy_success=9/50(18%) · ik_fails=0`. 정리하면
**"모델이 plausible·reachable 하게 움직이는데 과제만 못 푼다."** 그럼 왜?

## 2. 의심이 바뀌어 간 과정

### 의심 1 — "데이터나 변환 과정이 잘못된 거 아냐?"

EE(손끝) 좌표 변환·정규화 같은 데 숨은 버그를 의심했다. 그런데 **새 실험 없이도** 이미
가진 증거로 반박됐다: 변환은 이미 다른 게이트에서 검증돼 있었고, 학습이 정상적으로
끝났다는 것 자체가 "데이터·정규화는 자기 일관적"이라는 뜻이었다. → **데이터 결백.**
화살은 "데이터"가 아니라 "모델/방법"으로 향했다. *(이 부분은 처음에 내가 과장했다가
사용자 반박으로 정정 — 재발 방지 규율은 CLAUDE.md 협업 규칙 "추론 규율".)*

### 의심 2 — "색깔을 못 골라서 그런 거 아냐?" → **반전**

가장 그럴듯한 가설이었다. 학습할 때 모델의 "눈"과 "언어" 부분을 얼리고 손동작만
가르쳤으니(=head만 학습), "빨강이라는 단어 → 화면의 빨간 큐브" 연결을 못 배웠을 수 있다.
색을 랜덤으로 찍는다면 3개 중 1개(33%)만 맞을 거고, 집기 성공률과 곱하면 ~18%가 나온다.
계산이 딱 맞았다.

**그래서 측정했다 — "아무 큐브나 집었나" vs "시킨 색을 집었나"를 따로 셌다.**
(방법: 시뮬레이터 안에서 큐브 3개 각각이 잡혔는지·들렸는지 직접 들여다보고, 집어 든
큐브의 색을 목표색과 대조. `ThreeColoredCubesEnv`가 WSL eval 안에서 `base.cubes` 3개를
노출하고, 색 순서는 sidecar `label_metadata.target_id`로 매핑.)

결과는 **반전**이었다:

> **집은 큐브의 94%가 정확히 목표색.** 색깔은 거의 안 틀렸다.

색을 모르면 33%여야 하는데 94%다. → **언어/색 이해는 병목이 아니다.** 진짜 문제는
따로 있었다: **아무 큐브나 집어 드는 것 자체가 34%밖에 안 됐다.** 못 집은 경우, 손은
큐브에서 평균 **8cm 떨어진 데서** 멈췄다(성공할 땐 1cm). → 병목은 **reaching**.

*(이 가설을 측정 없이 믿고 비싼 풀-학습으로 갔으면 — 언어는 멀쩡한데 언어부를 손봐서
헛돈을 쓸 뻔했다. 측정이 막았다.)*

### 의심 3 — "16스텝을 한 번에 실행해서 도중에 빗나가나?" → 반증

모델은 한 번 보고 **앞으로 16스텝**의 손동작을 예측한다. 우리는 그 16개를 통째로
실행한 뒤에야 다시 카메라를 봤다. "눈 감고 16발짝 걷는" 셈이라, 도중에 살짝 어긋나면
못 고치고 8cm 빗나가는 게 아닐까?

검증: **얼마나 자주 다시 보는지(action_steps)만 16 → 8 → 4로 줄여** 재평가(학습은 그대로,
seed=0 고정으로 동일 50ep 위에서 페어 비교). 자주 보면 좋아져야 가설이 맞다. 결과:

| 다시 보는 주기 | 아무 큐브나 집기 | 성공률 | 색 정확도 |
|---|---|---|---|
| 16 (통째 실행) | 34% | 32% | 0.94 |
| 8 | 8% | 8% | 1.00 |
| 4 | 2% | 2% | 1.00 |

**자주 볼수록 오히려 더 나빠졌다** (단조 감소 — 노이즈로 보기엔 너무 일관). → 드리프트
가설 반증. 알고 보니 이 모델은 한 번에 길게 움직일수록 일관되고, 자주 다시 계획하면
**매번 살짝 다른 계획을 새로 뽑아 덜덜대다**(diffusion jitter) 못 집는다. 즉 **16(=호라이즌
최대)이 이미 최적**이고, 실행 방식으로는 못 고친다. (색은 4/4·1/1=100% → 의심 2 재확인.)

### 증거로 후보 정리 (✓남음 / ✗배제)

세 의심을 거치며, "그럴듯했던" 후보들이 가진 증거만으로 어떻게 갈렸는지:

| 후보 | 판정 | 근거 |
|---|---|---|
| **A. head-only가 언어 grounding 못 함** | ✗ 반증 | correct_color 94% → 색은 거의 완벽히 고름(랜덤 33%가 아님) |
| B. 정규화/relative_stats 누락 | ✗ 배제 | 학습 정상 완료(loss 수렴)+serve 동일 체크포인트 = 정규화 자기일관. 깨졌으면 학습이 안 떴음 |
| C. Path A(절대 rot6d) 미검증 "갭" | ✗ 배제(중복) | 저장 액션=`pose[t+1]`, `ee_verify` 타깃=재구성된 `pose[t+1]`(recon~0), rot6d round-trip~3e-7 → 같은 pose·같은 IK. ee_verify ~100%면 Path A도 ~100% |
| D. train/eval 이미지 도메인 갭 | 약(보류) | frozen visual이라 못 메움 |
| E. rot6d↔GR00T 내부 SE(3) 규약 | 약 | `ik_fails=0`이 gross corruption을 약하게 반증 |
| F. replan 거칢(action_steps=16) | ✗ 반증 | 16→8→4 단조 악화 (의심 3) |
| **reaching/grasping 정밀도** | ✓ **남음** | any_pick 34% + 실패 시 tcp→target 8.4cm. "뭘 집을지는 아는데 손이 못 닿음" |

→ 남은 결론: **병목은 모델이 손을 정밀하게 못 가져가는 것** 그 자체.

## 3. "이게 진짜 모델 문제 맞아?" — 돈 쓰기 전 공짜 검증 둘

비싼 재학습에 들어가기 전에, "8cm 빗나감"이 정말 모델 탓인지 **혹시 우리 측정이
잘못된 건 아닌지**부터 공짜로 확인했다 (둘 다 로컬 WSL, 무과금).

### 검증 ① 채점이 정직한가 (다트판이 아니라 답안지 점검)

학생(모델)을 채점하는데 **내 답안지가 틀렸으면** 학생이 맞아도 틀렸다고 나온다.
모델 출력 → 실제 로봇 움직임 사이엔 변환 단계가 여럿이라, 거기 버그가 있으면 모델을
억울하게 깎는다.

**확인법(모델 없이)**: 우리에겐 **100% 성공하는 정답 시범 행동**이 있다. 그걸 평가가
쓰는 **똑같은 변환 사슬**(기록 절대-EEF → rot6d→quat → IK → step)에 넣어 모델 없이 재생.

> 결과: **재현율 100% (50/50).** 정답을 넣으니 100% 성공.

→ **채점(변환·좌표·IK)은 정직하다.** 모델이 실패한 건 진짜 모델 탓. (덤으로 **데이터
품질도 완벽**임이 확인됐다.)

### 검증 ② 빗나감이 "한쪽으로 일정"이냐 "제멋대로"냐

다트가 **전부 오른쪽 8cm**에 꽂히면 → 가늠자(좌표)가 틀어진 버그(공짜 수정). **사방으로
흩어지면** → 그냥 손이 부정확한 것(학습 필요).

실패한 경우의 빗나간 거리를 보니 **0~13cm로 넓게 흩어져 있었다**(std 3.3cm, 한 값에 안
뭉침). → 고정 버그가 아니라 **진짜 부정확(산포)** 쪽. *(단 거리만 봤고 방향까진 못 봐서,
다음 실행 때 방향까지 기록되게 `miss_vec` 로깅을 보강해 뒀다.)*

**검증 종합**: 채점 정직 + 데이터 완벽 + 빗나감 산포 → **8cm 빗나감은 측정 착시가 아니라
진짜 모델 한계.** 이제 학습이 정당화된다.

## 4. "그럼 뭘 고쳐야 하나?"

### 왜 정밀하게 못 가나 — 모델이 보는 것과 모르는 것

모델은 매 순간 **세 가지**를 입력으로 받는다: 📷 카메라 영상 / 🦾 **자기 손 위치**(관절·손끝
좌표, 센서로 *정확히* 주어짐) / 💬 언어 지시.

핵심: **자기 손이 어디 있는지는 모델이 정확히 안다.** 화면에서 추정해야 하는 건
**큐브가 3D로 어디 있는가**뿐이다(큐브 좌표는 입력에 없고 영상에만 있음). 그런데
**시범 데이터는 큐브의 정확한 좌표를 알고 만들어졌고**, 모델은 그 좌표를 **카메라 픽셀로
눈대중 추정**해야 한다. 게다가 카메라가 **고정된 하나**라 깊이(멀고 가까움) 추정이 애매
→ 큐브를 cm 단위로 잘못 짚고 → 손이 그만큼 빗나간다.

### "더 학습해도 0개선"의 의미 — 눈이 얼어 있었다

여기서 사용자가 날카로운 질문을 했다: *"인지 문제라면 더 학습하면 나아져야지, 왜 0개선?"*

답: 우리 학습은 모델의 **"눈"(시각 인코더)을 얼려놨다.** 우리가 늘린 학습은 눈은 그대로
두고 **손동작 부분만** 더 돌린 것. 눈이 얼었으니 **인지는 아무리 학습해도 안 변한다.**
그래서 평탄했던 것 = **천장이 "얼린 눈"에 있다**는 증거.

### 공식 문서는 뭐라 하나 (GR00T 가이드)

NVIDIA의 GR00T fine-tune 가이드를 찾아봤다:

- **기본 설정 = 눈·언어 백본을 둘 다 얼리고, 연결층(projector) + 동작 head만 학습.**
  → 이게 **정확히 우리가 한 것**이다(표준을 그대로 따랐고, 빠뜨린 실수는 없었다).
- GR00T는 일부러 **비전에 의존하도록** 설계됐다(학습 때 자기 위치 정보를 가끔 가려 영상을
  보게 만듦). 즉 "얼린 눈이 충분히 잘 본다"는 전제를 깐다.
- **그 전제가 안 통할 때의 공식 처방 = `tune_visual`(눈 녹이기) / `tune_llm`.**

→ 우리가 데이터로 도달한 결론("얼린 눈이 천장")과 NVIDIA 공식 처방이 일치한다.

### 마지막 검토 — 공식 정밀 레시피는 손목 카메라 (2026-06-29)

처방을 확정하기 전, NVIDIA 공식 + 커뮤니티 가이드를 정독해 "우리 세팅이 표준과 어디서
갈리나"를 무과금으로 확인했다:

| 항목 | 공식 권장 | 우리 |
|---|---|---|
| **카메라** | **손목(그리퍼) + 정면 2개** (Seeed "손목 1 + 책상 1 권장"; NVIDIA 예제 wrist+front) | **고정 base_camera 1개뿐** |
| 파인튜닝 범위 | 기본 = 액션 헤드+projector (비전·LLM frozen); `tune_visual` 선택 | 동일 (head-only) |
| 정규화 레시피 | state dropout + color jitter로 비전 의존 유도 | (GR00T 내부 동일) |
| LR / 스텝 / batch | 1e-5 / 10k~20k / 16~32 | 20k / batch 16 — **같은 급** |

**결정적 발견**: 공식 정밀 레시피는 **손목 카메라**를 쓴다. 우리 병목(단일 고정 카메라로
큐브 3D 위치를 정밀하게 못 잡음)은 정확히 손목 카메라가 푸는 문제 — 그리퍼가 다가갈수록
큐브를 코앞에서 봐 깊이 모호성을 제거한다.

### 그래서 두 갈래의 fix — 그리고 무엇을 골랐나

| | 무엇 | 공략하는 원인 | 비용 |
|---|---|---|---|
| **B** | **눈 녹여 재학습**(visual fine-tune, warm-start) | 얼린 눈이 정밀 위치를 못 뽑음 | 학습 1회(싸고 빠름) |
| **D** | **손목 카메라 추가** | 단일 고정 카메라라 깊이가 애매 | env+데이터 재생성+학습(큼) |

처음엔 "head-only로 천장"이 보였으니 **B를 1순위**로 적었다. 그러나 위 카메라 재검토에서
**진단(단일 고정 카메라 한계)과 가장 정합하는 건 D**임이 드러났다 — B는 같은 카메라 1개로
"눈"만 녹이는 것이라 **단일 고정 캠의 깊이 모호성이 구조적으로 남아** 천장이 낮을 위험이
있고, D는 NVIDIA 표준 정밀 레시피와 정렬되며 reaching 정밀도 천장 자체를 올린다.

→ **D 채택.** (B는 "D를 해도 천장이면" 가는 카드로 보류.)

## 5. 결과 — 가설 검증됨 ✅ (2026-06-30)

D(손목 카메라)를 범용·하위호환 방식으로 파이프라인에 추가하고(구현·검증 상세는
`wrist-camera-impl.md`), 데이터 재생성(HF v2) → 재학습 2회 → 평가했다. 진단이
**결정적으로 검증**됐다. 50ep, seed=0, action_steps=16 (전부 동일조건):

| | v1-s3000 (1캠) | v1-s20000 (1캠) | v2-s3000 (손목캠) | **v2-s20000 (손목캠)** |
|---|---|---|---|---|
| 성공률 | 0.20 | 0.32 | 0.48 | **0.54** |
| any_pick | — | 0.34 | 0.48 | 0.54 |
| correct_color | — | 0.94 | 1.00 | **1.00** (27/27) |
| 못 집은 실패 손→큐브 빗나감 | — | **8.4cm** | 2.0cm | **1.5cm** |
| 성공 시 손→큐브 | — | 1.1cm | 0.9cm | 0.9cm |
| 학습 스텝 | 3000 | 20000 | 3000 | 20000 |

**① 손목캠 효과 (같은 스텝, 1캠→손목캠):**
- 3000스텝: 0.20→**0.48** (2.4배). 20000스텝: 0.32→**0.54** (1.7배).
- 손목캠 v2-s3000(3k)이 **1캠 20000스텝(0.32)보다도 높다**(1/7 학습량) — reaching 천장 자체가 올라감.
- **결정적 증거**: 진단의 "단일 고정 카메라 깊이 모호성 → 8cm 빗나감"이 손목캠으로 **8.4→1.5cm**
  급감. 못 집은 실패도 gross 8cm가 아니라 1.5~2cm 근접 실패(최소 0.3cm).

**② 스텝↑ 효과 (손목캠, 3k→20k):**
- 성공 0.48→**0.54** (+6%p), 근접 실패 빗나감 2.0→**1.5cm** (더 좁혀짐).
- 단 v1의 3k→20k(+12%p)보다 증가폭이 작다 — 손목캠은 3k에 이미 0.48로 빨리 학습해 **조기에
  천장 근처**. 남은 1.5cm 근접 실패는 스텝보다 **해상도(128²)·카메라 각도**가 다음 레버일 가능성.

**③ 공통 패턴 (손목캠):** 집은 건 **전부 성공**(any_pick=success), 색 **100%** → 유일한 잔여 실패
모드 = "1.5cm 근접 실패". 닿기만 하면 무조건 잡고 색도 맞춘다.

주의: 50ep 단발이라 확산 정책 분산으로 절대값 CI는 유의하나, 1캠↔손목캠 격차(빗나감 8.4→1.5cm,
성공 0.32→0.54)는 노이즈 초과 — 신호 견고. (모델: `@v2-s3000`(0.48)·`@v2-s20000`(0.54), MANIFEST 기록.)

## 부록 B — 측정 방법 (어떻게 그 숫자를 뽑았나)

- **성공**: 시뮬 환경의 `evaluate()` = 목표색 큐브를 잡고 테이블 위로 들었나(z>0.06).
- **색 정확도**: 매 스텝 큐브 3개 각각 `is_grasping`+높이를 직접 읽어, 집어 든 큐브 색을
  목표색과 대조. "집은 것 중 목표색 비율".
- **채점 정직성 게이트**: `scripts/_wsl_abs_eef_replay.py` — 기록된 절대-EEF 행동을 평가와
  똑같은 디코드(rot6d→quat→IK→step)로 모델 없이 재생, 재현율 확인. (WSL, 무과금)
- **빗나감 방향**: `_wsl_gr00t_eval` 의 `miss_vec`(손−큐브 벡터). 다음 실행부터 기록됨.
- **부수 계측**(같은 롤아웃에서 공짜): 에피소드별 ik_fails, 큐브별 max lift, 최종
  tcp→target 거리, 사용 스텝, 뽑은 색 → `*.eval_diag.jsonl`.

## 부록 C — 한 줄 용어

- **head만 학습**: 모델의 눈·언어는 얼리고, 동작 생성부(projector+diffusion head)만 학습.
- **visual fine-tune (tune_visual)**: 얼린 눈(시각 인코더)도 같이 학습 — 인지를 바꾸는 레버.
- **action_steps**: 한 번 보고 예측한 16스텝 중 *실제로 몇 개를 실행하고 다시 볼지*(실행 노브).
- **warm-start**: 바닥부터가 아니라 이미 학습된 모델(v1-s20000)에서 이어 학습.
- **diffusion jitter**: 확산 정책이 매번 살짝 다른 계획을 뽑아, 자주 재계획하면 동작이 떨림.
